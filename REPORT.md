## 1. Реализация Redis-кэша
Реализовано кэширование двух read-heavy endpoint:
- каталог товаров (агрегат по `order_items`);
- карточка заказа (`order + items`).

Ключи:
- ключ каталога: `catalog:v1`;
- ключ карточки заказа: `order_card:v1:{order_id}`.

TTL:
- `catalog:v1` — 60 секунд;
- `order_card:v1:{order_id}` — 60 секунд.

Логика cache hit/miss:
- при `use_cache=true` сервис сначала читает Redis;
- если ключ найден (hit), возвращаются данные из Redis;
- если ключа нет (miss), данные читаются из PostgreSQL и пишутся в Redis с TTL;
- при `use_cache=false` чтение выполняется напрямую из БД (без использования кэша).

## 2. Демонстрация неконсистентности (намеренно сломанный сценарий)
Сценарий реализован endpoint `POST /api/cache-demo/orders/{order_id}/mutate-without-invalidation`.

Шаги:
1. Прогрев кэша: `GET /api/cache-demo/orders/{id}/card?use_cache=true`.
2. Изменение заказа в БД: вызов `mutate-without-invalidation` с новым `total_amount`.
3. Повторный `GET .../card?use_cache=true` возвращает stale-значение из Redis.
4. Контрольный `GET .../card?use_cache=false` показывает актуальное значение из БД.

Результат stale cache:
- в кэше остаётся старый `total_amount`;
- в БД уже записан новый `total_amount`;
- пользователь при чтении через кэш видит устаревшие данные.

## 3. Починка через событийную инвалидацию
Где генерируется событие:
- в `POST /api/cache-demo/orders/{order_id}/mutate-with-event-invalidation` после успешного `UPDATE orders`.

Где обрабатывается событие:
- в `CacheInvalidationEventBus.publish_order_updated(...)`.

Какие ключи инвалидируются и почему:
- `order_card:v1:{order_id}` — чтобы следующая выдача карточки заказа не была stale;
- `catalog:v1` — чтобы пересобрать агрегаты каталога при следующем чтении и исключить устаревший агрегированный срез.

Проверка тестом:
- `test_cache_event_invalidation.py` подтверждает удаление ключа карточки из Redis и получение уже обновлённых данных после повторного чтения.

## 4. Rate limiting endpoint оплаты через Redis
Policy:
- `5` запросов за `10` секунд на subject.

Ключ лимита:
- `rate_limit:pay:{subject}`;
- `subject` = `X-User-Id`, если передан, иначе `client IP`.

Поведение при превышении:
- если счётчик `INCR` в окне превышает лимит, endpoint возвращает `429 Too Many Requests`.

Заголовки:
- `X-RateLimit-Limit`;
- `X-RateLimit-Remaining`;
- `X-RateLimit-Reset`;
- для заблокированного запроса дополнительно `Retry-After`.

Проверка тестом:
- `test_payment_rate_limit_redis.py`: первые 5 запросов проходят, 6-й получает `429`, значения `X-RateLimit-*` корректны.

## 5. Бенчмарки RPS до/после кэша
- Endpoint: `/api/cache-demo/catalog`
- Без кэша: RPS `485.29`, p95 latency `~230.51 ms`*
- С кэшем: RPS `901.33`, p95 latency `~111.21 ms`*
- Изменение: `+85.74%` по RPS

- Endpoint: `/api/cache-demo/orders/{order_id}/card`
- Без кэша: RPS `475.61`, p95 latency `~248.68 ms`*
- С кэшем: RPS `839.83`, p95 latency `~119.49 ms`*
- Изменение: `+76.59%` по RPS

\* Для `wrk` в использованной команде зафиксирована `Avg Latency` (без отдельного p95 в стандартном выводе), поэтому в отчёте приведена latency по `Avg` как ориентир для сравнения режимов.

## 6. Выводы
1. Для read-heavy endpoint (`catalog`, `order card`) Redis-кэш даёт существенный прирост пропускной способности (в текущем замере +76–86% RPS) и снижает задержки.
2. Простое добавление кэша без инвалидации приводит к stale data, даже если данные в БД уже обновлены корректно.
3. Инвалидация сложнее кэширования, потому что нужно точно знать, какие ключи зависят от изменённой сущности, и синхронно/событийно их очищать.
4. Rate limiting нужен даже при бизнес-валидациях: он отсекает всплеск повторов и защищает сервис на входе, до дорогих операций в БД.
5. Комбинация кэша + событийной инвалидации + rate limiting даёт одновременно производительность, консистентность чтения и устойчивость под нагрузкой.
